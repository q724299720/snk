# 名称识别方案说明

## 文档职责

本文档记录名称搜索、OCR 文本辅助、手动创建兜底的优先级、流程和实现约束。

## 识别优先级

对“美食 / 零食”场景，识别链路固定为：

1. 名称输入搜索
2. OCR 文本提取辅助
3. 手动创建

原因：

- 名称搜索是当前 MVP 的主路径，链路最短、最稳定
- OCR 适合作为输入辅助，减少手动敲字成本
- 手动创建用于承接长尾条目和冷启动空缺

## 名称识别流程

1. 用户输入食物名称、品牌或口味词
2. App 实时触发文本搜索
3. 服务端返回候选食物列表
4. 候选卡片展示名称、图片、全站平均分和基础分类信息
5. 用户确认条目后进入“记一笔”页面
6. 未命中时，用户可直接手动创建 `pending FoodItem`

## OCR 文本辅助流程

1. 用户拍照或从相册选择图片
2. App 在本地执行 OCR，提取品牌词、品类词、口味词
3. OCR 结果回填到主搜索框
4. App 自动复用同一套文本搜索链路
5. 用户仍在同一套候选列表中完成确认或手动创建

## 搜索结果展示策略

名称识别的候选列表固定优先展示以下信息：

1. 名称
2. 图片
3. 全站平均分
4. 基础分类、品牌等辅助信息

推荐规则：

- 有图片时优先展示图片，帮助用户更快确认目标条目
- 有评分时展示全站平均分，作为辅助识别信号
- 无评分时展示“暂无评分”，不阻断结果展示
- 无图片时展示占位图，不影响进入记录流程

## OCR 搜索策略

MVP 阶段对 OCR 文本不采用复杂 NLP，但也不能只做原样整串搜索。

推荐策略：

1. 保留 OCR 原始文本串
2. 做轻量归一化
3. 提取基础拆词组合
4. 将三类结果一起用于搜索召回

### 轻量归一化规则

- 去空格
- 去明显噪声字符
- 合并重复词
- 尽量保留品牌词、品类词、口味词

### 搜索输入建议

- 原串：如 `乐事 薯片 黄瓜味`
- 归一化串：去噪、去空格后的版本
- 拆词组合：如 `乐事 / 薯片 / 黄瓜味`

这样做的目标是提升 OCR 辅助输入场景下的命中率，而不引入过重的文本处理链路。

## 搜索失败兜底

- 搜索或 OCR 回填后仍未命中时，允许用户手动输入名称
- 服务端创建 `FoodItem` 候选条目，来源标记为 `user_generated`
- 条目默认 `pending`，仅创建者可见
- 审核通过后进入公共搜索结果
- 当搜索结果为空或候选不可靠时，当前页固定展示“手动创建”入口
- 手动创建入口只在空结果或候选不可靠时显示，不作为始终常驻入口
- 当前已落地两条入口：文本搜索空结果页、OCR 文本召回无命中页
- 手动创建成功后，客户端直接携带返回的 `pending FoodItem` 进入“记录创建”页，不要求用户再做一次搜索

## 用户纠错处理

当用户认为名称搜索或 OCR 辅助结果不正确时：

1. 允许用户改绑当前 `FoodRecord` 到正确的 `FoodItem`
2. 记录一条纠错信号，供后台排查条目归并、图片样本或搜索策略问题
3. MVP 阶段先保证“可追溯”，后续再决定是否进入自动学习闭环

## 手动创建中的候选提示

当用户已经开始手动创建 `pending FoodItem` 时：

1. 后续若发现可能匹配的候选条目，只作为提示展示
2. 不自动跳转、不自动覆盖、不自动中断当前输入
3. 是否改选已有条目由用户自己决定

## 候选不可靠判断

MVP 阶段对“是否需要展示手动创建兜底入口”的判断，采用前后端结合策略：

1. 后端返回候选质量信号
2. 前端结合当前场景决定是否展示兜底入口

推荐规则：

- 无结果：直接展示手动创建入口
- 有结果但后端标记为低置信 / 低质量：展示手动创建入口
- 有稳定候选且质量达标：不展示手动创建入口

## MVP 落地策略

- 名称搜索 + OCR 文本提取辅助 + 自建食物资料库

优点：

- 起步快
- 研发投入可控
- 比多入口识别链路更易维护和解释

## 进阶策略

- 名称搜索 + OCR + 聚合评分排序 + 更强的搜索召回策略

## 成熟策略

- 名称搜索排序优化
- 纠错反馈闭环
- 热门条目图片与评分资产沉淀

## 相关实现约束

- OCR 不直接跳独立识别页
- OCR 结果统一回填主搜索框
- 搜索结果必须复用同一套名称搜索链路
- 搜索结果卡片必须优先展示图片和全站平均分
- 本地 OCR 成功时，不要再触发独立的图片识别任务
- 搜索结果空或不可靠时，应回退到“手动创建”入口，而不是静默失败
- 用户手动纠错时，应修正当前记录并保留纠错信号
- 自研模型不是 MVP 阶段目标
