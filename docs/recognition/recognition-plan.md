# 识别方案说明

## 文档职责

本文件记录条形码、OCR、图像识别的优先级、流程、兜底规则和演进路线。

## 识别优先级

对“零食 / 包装食品”场景，识别链路固定为：

1. 条形码
2. 本地 OCR 文本搜索
3. 服务端 OCR 兜底
4. 图像识别 / 以图搜图

原因：

- 条形码最精准
- OCR 对相似包装差异口味更稳
- 图像识别更适合作为自然食物识别或最终兜底

## 图片识别流程

1. 用户拍照或从相册选择图片
2. App 做条形码扫描、本地 OCR、基础质量校验
3. 命中条形码则直接走精确检索
4. 无条形码但本地 OCR 有效时，直接做文本搜索召回
5. 本地 OCR 失败或质量差时，走服务端 OCR 兜底
6. 前述链路都不足时，才走图像识别 / 以图搜图
7. 服务端返回候选食物列表、置信度、相似图
8. 用户确认结果或手动修正
9. 服务端保存记录并更新检索索引

## OCR 搜索策略

MVP 阶段对 OCR 文本不采用复杂 NLP，但也不能只做原样整串搜索。

推荐策略：

1. 保留 OCR 原始文本串
2. 做轻量归一化
3. 提取基础拆词组合
4. 将三类结果一起用于搜索召回

### 轻量归一化规则

- 去空格
- 去明显噪声字符
- 合并重复词
- 尽量保留品牌词、品类词、口味词

### 搜索输入建议

- 原串：如 `乐事薯片黄瓜味`
- 归一化串：去噪、去空格后的版本
- 拆词组合：如 `乐事 / 薯片 / 黄瓜味`

这样做的目标是提升包装食品 OCR 场景下的命中率，而不引入过重的文本处理链路。

## 搜索 / 识别失败兜底

- 搜索、扫码、OCR、图像识别均未命中时，允许用户手动输入名称
- 服务端创建 `FoodItem` 候选条目，来源标记为 `user_generated`
- 条目默认 `pending`，仅创建者可见
- 审核通过后进入公共搜索结果

## MVP 落地策略

- 条形码 + OCR + 第三方识别 API + 自建食物资料库

优点：

- 起步快
- 研发投入可控
- 识别体验稳定性高于纯视觉路线

## 进阶策略

- 条形码 + OCR + 第三方识别 + 自建向量检索双路召回

## 成熟策略

- 自建视觉检索
- 纠错反馈闭环
- 样本回流训练

## 相关实现约束

- 本地 OCR 成功时，不要无条件上传图片走服务端 OCR
- 服务端 OCR 仅作兜底或增强
- 图像识别不应先于条形码和 OCR 触发
- 自研模型不是 MVP 阶段目标
