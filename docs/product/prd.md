# 产品需求说明

## 文档职责

本文件记录产品定位、目标用户、MVP 范围、扩展范围与冷启动策略。

## 产品目标

打造一个安卓端 App，面向“美食 / 零食发现、识别、记录、分享”场景，核心能力如下：

- 用户可通过输入名称搜索美食或零食
- 用户可通过扫码、OCR 或图片识别定位食物
- 用户可对食物进行评分、备注、打标签
- 所有业务数据保存在远程服务器
- 手机端负责采集、搜索、提交与展示
- 服务端负责数据存储、识别编排、审核与运营控制

## 目标用户

- 喜欢记录日常饮食的人
- 喜欢尝试零食并做评分的人
- 想通过图片快速识别食物的人
- 想记录“吃过什么、喜不喜欢、是否复购”的人

## MVP 范围

### 客户端 MVP

- 游客模式 / 匿名本地用户
- 首页最近记录 / 推荐入口
- 文本搜索食物
- 条形码扫码录入零食 / 包装食品
- 拍照或上传图片识别食物
- 本地 OCR 提取品牌和品名
- 查看识别结果和候选项
- 创建记录、评分、备注、标签
- 离线草稿箱与弱网补传
- 查看个人历史记录

### 服务端 MVP

- 用户体系
- 食物基础资料库
- 冷启动数据导入与外部食品数据兜底
- 图片上传与存储
- 条形码检索
- OCR 兜底识别
- 图像识别任务接口
- 搜索接口
- 记录与评分存储
- 后台管理：食物条目管理、用户记录审核、标签管理

## 第二阶段扩展范围

- 社区分享与点赞评论
- 收藏清单 / 想吃清单
- 地点关联
- 营养信息展示
- 增强版 OCR 结构化提取
- 识别结果纠错与模型反馈闭环
- 多模态检索

## 冷启动与数据增长策略

这是最优先解决的问题。

- 首批导入常见零食、饮料、菜品基础数据，至少准备数千条可搜索数据
- 早期接入外部开放食品数据库作为兜底来源
- 搜索或识别失败时，允许用户手动创建记录
- 用户手动创建的食物条目以 `pending` 状态进入候选池
- 待后台审核后，条目才可全局可见
- 用 UGC 持续补足长尾食物和新品零食

## 产品边界原则

- 第一阶段不做复杂社交功能
- 第一阶段不自研识别模型
- 第一阶段优先做可闭环的数据流和记录体验
- 第一阶段优先游客模式，不先做完整登录体系
- 游客模式在技术上使用服务端匿名用户，不采用纯本地匿名模型
- 零食识别场景优先使用条形码和 OCR，不把纯视觉识别作为主路径
- MVP 阶段不引入独立的分享 / 帖子对象
- 若未来扩社区，分享对象应独立于 `FoodRecord` 设计
- MVP 首页不做个性化推荐算法
- 首页“推荐”仅指最近记录、快捷入口、常见分类和少量热门静态内容
