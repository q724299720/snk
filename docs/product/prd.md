# 产品需求说明

## 文档职责

本文件记录产品定位、目标用户、MVP 范围、扩展范围与冷启动策略。

## 产品目标

打造一个安卓端 App，面向“美食 / 零食发现、识别、记录、分享”场景，核心能力如下：

- 用户可通过输入名称搜索美食或零食
- 用户可通过图片和评分辅助确认搜索到的食物
- 用户可通过 OCR 提取文本并回填搜索
- 用户可对食物进行评分、备注、打标签
- 所有业务数据保存在远程服务器
- 手机端负责采集、搜索、提交与展示
- 服务端负责数据存储、识别编排、审核与运营控制

## 目标用户

- 喜欢记录日常饮食的人
- 喜欢尝试零食并做评分的人
- 想通过名称搜索快速确认食物的人
- 想记录“吃过什么、喜不喜欢、是否复购”的人

## MVP 范围

### 客户端 MVP

- 游客模式 / 匿名本地用户
- 首页最近记录 / 推荐入口
- 最近搜索（本地缓存）
- 清空最近搜索
- 实时名称搜索食物
- 搜索结果展示图片
- 搜索结果展示全站平均分
- 本地 OCR 提取品牌和品名并回填搜索
- 创建记录、评分、备注、标签
- 离线草稿箱与弱网补传
- 查看个人历史记录

### 服务端 MVP

- 用户体系
- 食物基础资料库
- 冷启动数据导入与外部食品数据兜底
- 图片上传与存储
- 搜索接口
- 搜索结果聚合评分返回
- 记录与评分存储
- 后台管理：食物条目管理、用户记录审核、标签管理

## 第二阶段扩展范围

- 社区分享与点赞评论
- 当前已落地最小公开分享：用户在创建记录时主动打开公开开关后，记录进入首页公开分享区；默认仍为私有
- 当前已落地最小公开评论：公开记录可查看最新评论并提交评论；私有记录不开放评论入口
- 收藏清单 / 想吃清单
- 地点关联
- 营养信息展示
- 增强版 OCR 结构化提取
- 识别结果纠错与模型反馈闭环
- 多模态检索

## 冷启动与数据增长策略

这是最优先解决的问题。

- 首批导入常见零食、饮料、菜品基础数据，至少准备数千条可搜索数据
- 早期接入外部开放食品数据库作为兜底来源
- 外部开放数据源的选择、运行时边界与批量导入规则，单独维护在 `docs/product/external-data-strategy.md`
- 搜索或识别失败时，允许用户手动创建记录
- 当搜索结果为空或识别无可靠候选时，当前页固定展示“手动创建”入口
- MVP 阶段手动创建入口文案优先采用带解释形式，如“没有找到？手动创建”
- 手动创建入口只在空结果或候选不可靠时显示，不始终占据搜索结果页
- 用户手动创建的食物条目以 `pending` 状态进入候选池
- 手动创建 `pending` 条目时，最小必填字段仅为 `name + item_type + category`
- 手动创建时一级分类必选，二级分类选填
- 手动创建时图片选填，不作为兜底创建前置条件
- 手动创建表单直接提供可选 `brand` 输入框
- 手动创建表单不暴露 `search_keywords`，由系统自动生成基础关键词
- 手动创建 `pending FoodItem` 成功后，自动回填当前记录并继续保存本次 `FoodRecord`
- 创建者在搜索结果里看到自己的 `pending FoodItem` 时，应有明确“待审核”标记
- 对创建者本人而言，搜索结果中 `approved` 条目默认排在前，`pending` 条目靠后展示
- 用户一旦进入手动创建流程，后续即使发现匹配候选，也只做可选提示，不自动打断当前输入
- 若 `FoodRecord` 绑定的是 `pending FoodItem`，则在最近记录中也应显示轻量“待审核”提示
- `pending FoodItem` 采用异步人工审核为主；若 1 天内无人审核，则进入自动审核
- 自动审核只做保守动作，如拒绝明显无效条目或高置信合并；不直接大范围自动批准
- 自动拒绝标准采用极保守策略，只拒绝明显垃圾数据
- “极短乱码”采用组合判断，不做简单长度一刀切
- 待后台审核后，条目才可全局可见
- 用 UGC 持续补足长尾食物和新品零食

## 产品边界原则

- 第一阶段不做复杂社交功能
- 第一阶段不自研识别模型
- 第一阶段优先做可闭环的数据流和记录体验
- 第一阶段优先游客模式，不先做完整登录体系
- 游客模式在技术上使用服务端匿名用户，不采用纯本地匿名模型
- 游客匿名身份在 MVP 阶段采用安装级生命周期，卸载重装后视为新游客
- MVP 阶段不做条码识别
- MVP 阶段不做以图搜图
- OCR 仅作文本提取辅助，不作为主识别链路
- MVP 阶段不引入独立的分享 / 帖子对象
- 若未来扩社区，分享对象应独立于 `FoodRecord` 设计
- MVP 首页不做个性化推荐算法
- 首页“推荐”仅指最近记录、快捷入口、常见分类和少量热门静态内容
- 首页中的“热门静态推荐”在 MVP 阶段采用手工配置来源
- 首页最近记录默认展示最近 `5` 条
- 首页正式最近记录与待上传草稿 / 失败重试项分区展示
