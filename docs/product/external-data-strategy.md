# 外部开放数据源接入策略

## 文档职责

本文件定义 `Phase 0 / Step 4` 的外部开放数据源选择原则、MVP 接入范围、运行时使用方式、批量导入方式和合规约束。

## 目标

外部数据源只用于缓解冷启动，不替代自有主库，不直接决定 App 端主搜索体验。

MVP 阶段要解决的是：

- 包装食品条形码命中不足时，有一个可靠兜底来源
- 后续可逐步补齐营养、品牌、图片等结构化信息
- 不把运行时链路绑死在不稳定的第三方接口上
- 不因为外部数据引入不可控的许可证或审核风险

## 选择原则

- 优先选择有官方文档、可持续访问、许可清晰的数据源
- 运行时优先条形码精确命中，不依赖外部全文搜索
- 客户端不直接请求第三方数据源，统一由服务端适配
- 先解决包装食品冷启动，不试图靠外部源覆盖全部菜品
- 外部导入结果必须经过本地字段映射和最小校验，不能原样透传

## MVP 结论

### 主结论

- 自建种子数据仍是主数据基线
- `Open Food Facts` 作为 MVP 阶段唯一优先接入的外部开放食品数据源
- `USDA FoodData Central` 仅保留为后续营养信息增强候选，不进入 MVP 运行时主链路
- 暂不接入未完成稳定性、文档和许可核验的其他第三方国内接口

### 原因

`Open Food Facts` 当前具备：

- 明确的官方 API 文档
- 条形码读取场景成熟
- 可下载的数据集
- 可复用的图片与 OCR 相关公开数据

但它也有明确限制：

- 数据由志愿者贡献，不保证完整和准确
- 新版服务端 API 不适合作为我们 App 的全文搜索主链路
- 图像与数据库内容存在不同许可证约束

因此，MVP 应把它当作“包装食品条码兜底 + 批量冷启动补数来源”，而不是“外部实时搜索引擎”。

## 数据源分工

### 1. 自建种子数据

作用：

- 承担 App 默认文本搜索
- 承担首批高频品牌和高频菜品覆盖
- 承担已知有效词、分类和标签基线

说明：

- 自建数据是主库基线，不因外部源可用而取消
- 首批样本和后续批量导入都必须映射到统一 `FoodItem` 模型

### 2. Open Food Facts

MVP 用途：

- 包装食品条形码兜底查询
- 批量补充高频零食 / 饮料基础数据
- 后续可选补充包装图、OCR 相关图像样本

明确不做：

- 不作为 App 运行时全文搜索主接口
- 不直接让客户端访问 Open Food Facts
- 不覆盖中式菜品、现制饮品、堂食餐食的主目录建设

### 3. USDA FoodData Central

保留用途：

- 后续水果、通用食材、营养信息展示增强
- 后续营养维度校验或补全

MVP 不采用的原因：

- 需要 API Key
- 更偏营养与通用食物数据，不是中文零食识别主场景
- 不适合作为中国包装零食的运行时主兜底源

## Open Food Facts 接入策略

### 运行时链路

服务端统一处理，客户端不直连第三方：

1. 后台或导入脚本基于人工维护的条码清单先查自有 PostgreSQL
2. 若本地未命中，且属于带条形码的包装食品数据补全场景，则由服务端调用 Open Food Facts 适配层
3. 适配层命中后先做字段归一化和最小校验
4. 校验通过则写回自有库，供后续名称搜索和 OCR 关键词召回使用
5. 校验失败或未命中则继续通过人工整理或用户手动创建补齐

约束：

- 只用于条形码精确查找，不接外部全文搜索
- 只在服务端 miss 时触发，不把第三方请求放到主路径最前面
- 结果入库后后续请求优先命中自有库，避免重复请求外部源
- 当前 Android MVP 不提供扫码触发的运行时外部查询

### 文本搜索策略

MVP 阶段文本搜索仍以自有 PostgreSQL 为主。

原因：

- Open Food Facts 官方文档明确：`/api/v2/search` 提供结构化搜索，`/api/v3/search` 尚未实现
- 官方同时说明其全文 / 关键词搜索能力不应作为新接入的首选主方案
- Open Food Facts 当前建议新接入优先使用 v3；若读取能力仅在 v2 可用，则由服务端适配层内部兼容 v2

因此项目结论为：

- 我们自己的名称搜索、OCR 文本搜索、别名搜索全部走自有库
- 外部源只负责补数据，不负责承担用户侧主搜索体验

### 批量导入策略

MVP 阶段分两种模式：

1. `按条形码拉取`
2. `按数据集筛选导入`

#### 按条形码拉取

适用场景：

- 已有品牌清单、条码清单
- 需要快速补齐样本
- 需要低风险、可控地扩容首批数据

策略：

- 维护人工精选条码列表
- 由导入脚本逐条调用 Open Food Facts
- 导入前做字段裁剪和校验

#### 按数据集筛选导入

适用场景：

- 后续需要更大规模补齐包装食品
- 需要离线筛选高频品牌或类目

策略：

- 优先下载官方数据集，在本地脚本中筛选品牌、类目和必要字段
- 先进入临时导入文件，再进入我们的 PostgreSQL
- 不直接把完整外部大表原样塞进主业务库

## 字段映射原则

Open Food Facts 导入到本项目时，至少映射：

- `barcode`
- `name`
- `brand`
- `category`
- `alias`
- `search_keywords`
- `cover_image_url`
- `source=external_api`

规则：

- `search_keywords` 仍由我们自己的规则生成和补充
- 外部分类不直接原样暴露给前端，需映射回本项目分类体系
- 缺失关键字段的外部结果不能直接成为高质量正式条目
- 条形码冲突时，以现有主库唯一性规则为准，不允许外部结果覆盖本地主记录

## 审核与质量策略

- 外部数据不是“天然可信”，只是“优先可用”
- 运行时通过条形码拉取的结果，只有在满足最小字段和唯一性规则时才允许入主库
- 对缺失名称、名称异常、分类无法映射、条形码冲突的数据，导入脚本必须跳过或打入人工处理清单
- 外部数据写回后仍保留 `source=external_api`，便于后续筛查、纠错和再清洗

## 合规与许可约束

### Open Food Facts

- 数据库数据采用 ODbL / Database Contents License 体系
- 产品图片受单独的开放许可约束，使用前需按官方条款核验
- 官方文档明确其数据由用户志愿贡献，不保证准确、完整或可靠
- 官方要求所有 API 调用必须携带自定义 `User-Agent`
- 官方对各 API 端点执行限流

项目约束：

- 必须在内部保留外部来源标记
- 不能把外部图片和数据库内容当作无条件自有资产处理
- 若未来要公开二次分发组合数据库，必须重新做许可证评估

### USDA FoodData Central

- 数据为 CC0 / Public Domain
- 需要保护 API Key，不得提交到仓库

项目约束：

- 若未来启用 FDC，API Key 只放服务端安全配置
- 不在任何文档、代码库、样例文件中写入明文 Key

## 缓存与稳定性策略

- 客户端不做第三方源缓存
- 服务端以 PostgreSQL 作为外部命中后的本地沉淀层
- 同一条码首次外部命中后，后续应直接命中本地库
- 对高频品牌和高频条码，优先使用离线批量导入预热，而不是依赖用户触发式回填

## 暂不启用的能力

- 不在 MVP 阶段接入 Open Food Facts 图片 OCR 数据集进行训练
- 不在 MVP 阶段接入 Robotoff 预测接口进入生产识别主链路
- 不在 MVP 阶段把 USDA 作为中文零食搜索补数源
- 不在 MVP 阶段接入未完成许可和稳定性审查的其他第三方国内接口

- 已明确 MVP 优先接入的外部数据源
- 已明确运行时链路与批量导入链路分离
- 已明确文本搜索不依赖外部 API
- 已明确许可、图片和 API Key 约束
- 已明确后续导入脚本应遵循的最小字段映射规则
