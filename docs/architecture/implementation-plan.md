# 模块化实施计划

## 文档职责

本文件定义唯一有效的开发顺序、阶段目标、交付物和增量测试要求。后续开发必须严格按本文件执行。

## 执行总原则

- 不允许跳阶段开发
- 不允许在未完成上一阶段验证前进入下一阶段
- 每个阶段必须有明确交付物
- 每个阶段完成后必须执行增量测试并记录结果
- 若阶段目标变更，必须先更新本文件和 `agents.md`

## Phase 0：冷启动数据准备

### 目标

为搜索、扫码、OCR 和识别提供可用的首批基础数据。

### 交付物

- 食物分类与标签体系
- 首批零食、饮料、菜品基础数据
- 外部开放数据源接入策略
- 数据导入脚本方案

### 子步骤

1. 固定食物分类与标签体系
2. 固定种子数据字段规范与校验规则
3. 整理首批品牌、条形码和菜品样本
4. 明确外部开放数据源接入策略
5. 编写导入脚本方案或脚本骨架

### 增量测试

- 抽样验证首批数据可被文本搜索命中
- 验证条形码字段可唯一或高置信命中目标条目
- 验证导入数据字段完整性

## Phase 1：后端基础与数据库迁移

### 目标

建立后端工程基础、数据库迁移机制、核心表结构和上传闭环。

### 交付物

- 服务端项目初始化
- PostgreSQL 连接与迁移框架
- 核心表结构迁移脚本
- 图片上传接口与对象存储接入
- 游客模式所需的最小身份标识能力（服务端匿名 `user_id`）

### 增量测试

- 迁移脚本可重复执行
- 本地 / 测试环境可完成建库与回滚
- 上传接口可成功写入对象存储并返回资源地址
- 匿名用户初始化后可获得稳定的服务端身份标识
- 同一安装周期内匿名身份稳定，卸载重装后视为新匿名用户

## Phase 2：安卓端基础记录闭环

### 目标

先完成不依赖智能识别的记录闭环。

### 交付物

- Android 项目初始化
- 游客模式基础流程
- 文本搜索页面
- 条形码扫描录入
- 记录创建页
- Room 草稿存储
- WorkManager 离线补传

### 增量测试

- 文本搜索到创建记录的全链路可用
- 条形码扫描可命中服务端条目
- 弱网 / 无网下可保存草稿
- 网络恢复后可自动补传
- 游客模式下新建记录可与同一匿名用户关联
- 草稿图片复制到 App 草稿目录后，原始相册路径变化不影响后续补传
- 草稿上传达到自动重试上限后，会进入 `failed` 状态并允许用户手动重试
- `failed` 草稿会向用户展示轻量分类后的失败原因

## Phase 3：识别链路接入

### 目标

完成“条形码 > 本地 OCR > 服务端 OCR > 图像识别”的分层识别链路。

### 交付物

- 本地 ML Kit OCR 集成
- OCR 文本直搜逻辑
- 服务端 OCR 兜底接口
- 图像识别任务接口
- 候选结果确认页
- 搜索 / 识别失败时的手动创建条目流程

### 增量测试

- 有条形码的样本优先走扫码成功
- 无条形码但文字清晰的样本可由本地 OCR 命中文本搜索
- 本地 OCR 失败时可回退到服务端 OCR
- 图像识别接口可返回候选结果
- 未命中时可手动创建 `pending` 条目

## Phase 4：后台治理与审核

### 目标

建立 UGC 审核、纠错治理和后台管理基础能力。

### 交付物

- 后台管理基础框架
- 食物条目管理
- UGC 待审核队列
- 报错 / 纠错处理流程
- 搜索与识别任务监控

### 增量测试

- 未审核条目仅创建者可见
- 审核通过后条目进入全局搜索
- 报错记录可累计并触发人工处理
- 后台可查看识别失败与异常任务
- `pending` 条目 1 天内优先等待人工审核，超时后可进入自动审核流程
- 自动审核仅执行保守动作，不直接大范围批准 `pending` 条目

## Phase 5：增强能力与优化

### 目标

在闭环稳定后再扩推荐、社区和高级检索能力。

### 交付物

- 相似食物推荐
- 社区分享能力
- 评论点赞
- 识别纠错闭环
- 搜索与向量检索优化

### 增量测试

- 推荐结果与用户历史记录相关
- 社区内容权限和审核生效
- 高并发下搜索与识别链路仍稳定

## 开发前检查单

每次准备开始一个新任务前，必须确认：

- 已阅读 `agents.md`
- 已确认当前任务所属阶段
- 已阅读本阶段关联文档
- 已明确本次增量的测试方式
- 已预留文档变更记录更新动作
