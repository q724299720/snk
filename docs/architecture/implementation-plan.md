# 模块化实施计划

## 文档职责

本文件定义唯一有效的开发顺序、阶段目标、交付物和增量测试要求。后续开发必须严格按本文件执行。

## 执行总原则

- 不允许跳阶段开发
- 不允许在未完成上一阶段验证前进入下一阶段
- 每个阶段必须有明确交付物
- 每个阶段完成后必须执行增量测试并记录结果
- 若阶段目标变更，必须先更新本文件和 `agents.md`

## Phase 0：冷启动数据准备

### 目标

为名称搜索、OCR 文本辅助和手动创建提供可用的首批基础数据。

### 交付物

- 食物分类与标签体系
- 首批零食、饮料、菜品基础数据
- 外部开放数据源接入策略
- 数据导入脚本方案

### 子步骤

1. 固定食物分类与标签体系
2. 固定种子数据字段规范与校验规则
3. 整理首批品牌、常用名称别名和菜品样本
4. 明确外部开放数据源接入策略
5. 编写导入脚本方案或脚本骨架

### 增量测试

- 抽样验证首批数据可被文本搜索命中
- 验证导入数据字段完整性

## Phase 1：后端基础与数据库迁移

### 目标

建立后端工程基础、数据库迁移机制、核心表结构和上传闭环。

### 交付物

- 服务端项目初始化
- PostgreSQL 连接与迁移框架
- 核心表结构迁移脚本
- `ReviewConfigWord` / `ReviewConfigWordAuditLog` 等审核词典基础表迁移脚本
- 图片上传接口与对象存储接入
- 游客模式所需的最小身份标识能力（服务端匿名 `user_id`）

### 增量测试

- 迁移脚本可重复执行
- 本地 / 测试环境可完成建库与回滚
- 审核词典配置表与追加式审计日志表可随迁移成功创建
- 上传接口可成功写入对象存储并返回资源地址
- 匿名用户初始化后可获得稳定的服务端身份标识
- 同一安装周期内匿名身份稳定，卸载重装后视为新匿名用户

## Phase 2：安卓端名称搜索与记录闭环

### 目标

先完成不依赖智能识别的记录闭环。

### 交付物

- Android 项目初始化
- 游客模式基础流程
- 名称搜索页面
- 实时名称联想搜索
- 记录创建页
- Room 草稿存储
- WorkManager 离线补传
- 当前仓库已落地首页最近记录默认展示最近 5 条
- 当前仓库已落地首页正式最近记录与待上传草稿 / 失败重试项分区展示

### 增量测试

- 名称搜索到创建记录的全链路可用
- 输入关键词后可返回可点击的候选条目
- 弱网 / 无网下可保存草稿
- 网络恢复后可自动补传
- 游客模式下新建记录可与同一匿名用户关联
- 草稿图片复制到 App 草稿目录后，原始相册路径变化不影响后续补传
- 草稿上传达到自动重试上限后，会进入 `failed` 状态并允许用户手动重试
- `failed` 草稿会向用户展示轻量分类后的失败原因
- 首页最近记录固定展示最近 5 条，不返回更多
- 首页正式最近记录与待上传草稿 / 失败重试项分区展示，且已同步草稿不重复出现在本地分区

## Phase 3：OCR 文本辅助与搜索结果增强

### 目标

完成“名称搜索 > OCR 文本提取辅助 > 手动创建”的名称识别链路，并让搜索结果具备图片与评分辅助确认能力。

### 交付物

- 本地 ML Kit OCR 集成
- OCR 文本回填搜索框
- OCR 文本搜索复用逻辑
- 搜索结果卡片展示图片与全站平均分
- 搜索 / OCR 失败时的手动创建条目流程

### 增量测试

- 输入名称后可返回带图和评分的候选结果
- OCR 提取文本后可自动回填搜索框并复用同一搜索链路
- 无图片或无评分条目仍可稳定展示候选卡片
- 未命中时可手动创建 `pending` 条目

## Phase 4：后台治理与审核

### 目标

建立 UGC 审核、纠错治理和后台管理基础能力。

### 交付物

- 后台管理基础框架
- 当前仓库已落地轻量静态后台页 `/admin/index.html`，用于操作治理概览、食物条目、审核词典和审计日志
- 当前仓库已落地可选后台 API Token 保护，配置 `SNK_ADMIN_API_TOKEN` 后 `/api/admin/**` 需携带 `X-SNK-ADMIN-TOKEN`
- 食物条目管理
- UGC 待审核队列
- 当前仓库已落地食物条目列表 / 详情接口，可按状态与关键词查看条目
- 当前仓库已在轻量后台页落地食物条目列表详情查看入口
- 当前仓库已落地食物条目报错清零接口，可作为后台纠错处理结束动作
- 当前仓库已落地食物条目报错明细记录与后台查询接口
- 当前仓库已落地食物条目合并接口，可将重复条目的历史记录迁移到保留条目，并将重复条目标记为 `rejected`
- 当前仓库已在轻量后台页落地合并目标预览入口，降低输入错误目标 ID 的治理风险
- 当前仓库已落地搜索接口的创建者 `pending` 可见性：未传 `userId` 只返回已审核条目，传入创建者 `userId` 可看到自己的待审条目
- 审核词典配置与启停管理
- 审核词典追加式审计日志查询
- 当前仓库已落地审核词典配置、编辑、启停和审计日志查询接口
- 当前仓库已在轻量后台页落地审核词典新增、编辑、启停和审计日志入口
- 报错 / 纠错处理流程
- 搜索、报错与审核状态监控
- 历史预留识别任务监控接口仅保留兼容，不作为当前 MVP 的新增任务主线
- 当前仓库已在轻量后台页落地历史识别任务列表与详情监控入口
- 统计报表
- 当前仓库已落地后台治理汇总概览接口，可查看条目、任务与词典的关键总量
- 当前仓库已落地自动审核读取启用的 `valid_food_word` 词典，用于保守识别极短乱码并保留有效短词
- 当前仓库已落地后台手动触发自动审核接口与页面按钮，用于立即扫描超时待审核条目

### 增量测试

- 未审核条目仅创建者可见
- 审核通过后条目进入全局搜索
- App 主搜索会携带当前游客 `userId`，用于展示自己创建的待审核条目；OCR 文本搜索复用同一参数
- 后台可按 `auditStatus`、`q`、`limit` 查看食物条目列表并打开详情
- 后台页面可点击条目“详情”并调用 `GET /api/admin/food-items/{foodItemId}` 展示完整管理视图
- 后台可在处理报错后将目标条目的 `reportCount` 清零
- 后台页面处理条目后会刷新当前条目详情与报错明细，避免显示旧报错状态
- 报错记录可累计、保留明细，并触发人工处理
- 后台可查看单个食物条目的报错用户、原因与提交时间
- 后台可将重复 / 错误食物条目合并到保留条目，历史记录需迁移到保留条目，重复条目不再进入全局搜索
- 条目合并目标必须是 `approved` 条目，禁止迁移历史记录到 `pending / rejected` 条目
- 后台页面可在合并前预览目标条目的 ID、名称、审核状态和报错数，避免误输入目标 ID
- 后台页面在未预览目标、预览目标与输入 ID 不一致、或预览目标不是 `approved` 时，必须阻止合并请求
- 后台页面在目标 ID 与当前条目 ID 相同时，必须阻止合并请求，避免无效自合并
- 后台页面合并目标 ID 输入变化后必须清空旧预览缓存，避免旧预览继续误导后续合并判断
- 后台页面合并成功后必须清空合并目标输入框与预览缓存，避免下一次操作误用旧目标
- 后台可查看报错条目、待审条目和审核配置状态
- 历史识别任务监控接口如继续保留，只做兼容性验证，不作为 App 当前链路验收项
- 后台页面可查看历史识别任务列表，并打开单个任务详情排查失败原因
- 后台可查看治理汇总概览，且数据口径与底层记录计数一致
- 后台识别任务总数需包含 `pending / processing / completed / failed` 等历史状态，避免遗留任务被低估
- 后台页面可展示历史识别任务总数以及 `pending / processing / completed / failed` 分状态数量
- 配置后台 API Token 后，缺少或错误 token 的后台 API 请求会被拒绝
- `pending` 条目 1 天内优先等待人工审核，超时后可进入自动审核流程
- 自动审核仅执行保守动作，不直接大范围批准 `pending` 条目
- 自动拒绝仅针对空白、纯符号、极短乱码等明显垃圾数据
- “极短乱码”需采用组合判断：长度短且不含有效词、且明显不像真实食物名
- 词典新增 / 编辑 / 启停后，对后续审核任务立即生效
- 自动审核每次运行需读取当前启用的 `valid_food_word` 词典，避免后台词典变更后仍按旧有效词判断
- 后台页面可手动触发一次自动审核，并展示扫描数、拒绝数、保留待审数与拒绝条目 ID
- 后台页面可直接编辑既有词条并调用 `PUT /api/admin/review-config-words/{wordId}`
- 每次词典 `create / update / enable / disable` 操作都会追加审计日志
- 重新启用词条时沿用原 `ReviewConfigWord` 主记录，而不是新建新记录

## Phase 5：增强能力与优化

### 目标

在闭环稳定后再扩推荐、社区和高级检索能力。

### 交付物

- 相似食物推荐
- 社区分享能力
- 评论点赞
- 识别纠错闭环
- 搜索与向量检索优化
- 当前仓库已落地基于同品牌 / 同分类的相似食物推荐接口与记录页推荐卡片
- 当前仓库已落地记录分享按钮与系统分享摘要，便于社区传播和内容外发
- 当前仓库已落地记录点赞计数接口与创建成功页的点赞按钮，作为互动能力的最小闭环

### 增量测试

- 推荐结果与用户历史记录相关
- 社区内容权限和审核生效
- 点赞按钮能在手机端直接更新记录点赞数
- 高并发下搜索与识别链路仍稳定
- 记录页可展示推荐结果，并支持切换到推荐条目继续记一笔
- 记录成功后可拉起系统分享面板并分享记录摘要
