# SNK

美食 / 零食分享与记录项目的核心语义定义，避免在商品、菜品、记录对象等概念上混用术语。

## Language

**FoodItem**:
系统中统一的可被搜索、识别、记录和审核的目录实体，既可以表示包装食品商品，也可以表示菜品或其他可食用对象。`item_type` 在 MVP 阶段仅保留 `packaged_product / dish / fruit` 三种核心语义类型，而不是和分类体系重复，其中 `fruit` 保留为独立类型而不并入 `dish`。对于 `packaged_product`，同一条形码视为同一目录实体；无条形码但明显属于包装食品的条目，仍归类为 `packaged_product`。包装饮料也归入 `packaged_product`；现制饮品归入 `dish`。对于 `dish`，先按通用菜名建统一条目，不按店铺拆分。`pending` 状态的条目仅创建者和审核后台可见，对其他普通用户不可见。用户手动创建 `pending FoodItem` 时，最小必填字段为 `name + item_type + category`。包装食品若只是新旧包装、封面或文案变化，不新建条目，而是在同一 `FoodItem` 下吸收图片、别名和关键词变化。
_Avoid_: ProductItem, DishItem, 商品条目, 菜品条目

**FoodItem Merge**:
当审核确认多个 `FoodItem` 实际表示同一个对象时，后台应支持条目合并。合并时历史 `FoodRecord` 应迁移到保留条目，被合并条目不再作为普通搜索结果继续存在。
_Avoid_: 只删错误条目, 只修未来不修历史

**FoodRecord**:
用户围绕某个 `FoodItem` 产生的一次具体发生的记录事件，表示一次吃过、买过、试过或评价过的行为，而不是长期汇总卡片。评分、备注、图片、价格、地点等都附着在这次事件上。正式提交到服务端的记录必须绑定一个 `FoodItem`，但客户端本地草稿允许在识别完成前暂时未绑定。若用户在兜底流程中手动创建了 `pending FoodItem`，系统应自动将其回填到当前 `FoodRecord`，继续本次保存流程。`FoodRecord` 默认仅自己可见，公开应当是后续用户主动触发的动作。`rating` 表示本次事件体验评分，而不是用户对该食物的长期总评分。
_Avoid_: 长期评价卡片, 收藏卡片, 食物档案

**Share / Post**:
MVP 阶段暂不引入独立的分享内容对象。社区化分享不是当前记录闭环的一部分，若未来需要社区能力，再单独设计分享对象，而不是提前让 `FoodRecord` 承担帖子语义。
_Avoid_: 公开记录帖, 社区记录卡

**Home Recommendation**:
MVP 阶段首页中的“推荐”不表示个性化算法推荐，而是最近记录、快捷入口、常见分类和少量热门静态内容的组合。它的目标是帮助用户更快继续记录，而不是做复杂推荐系统。
_Avoid_: 个性化推荐流, 推荐算法首页

**Guest Mode**:
MVP 第一阶段优先支持游客模式，但游客在技术上仍对应一个服务端匿名用户标识，而不是纯本地匿名。这样记录、补传和远程数据存储都能围绕稳定的匿名 `user_id` 运转。该匿名身份在 MVP 阶段采用安装级生命周期：同一安装周期内保持稳定，卸载重装后视为新的匿名用户。正式账号登录不是当前首要验证对象，应在基础闭环稳定后再补充。
_Avoid_: 先上完整认证体系, 首发强制注册, 纯本地匿名

**OCR Search Normalization**:
OCR 识别文本在进入搜索前，MVP 阶段应采用轻量归一化，而不是只做原样整串搜索。系统应同时保留原串、归一化串和拆词结果参与检索，以提升包装食品命中率。
_Avoid_: 仅原串搜索, 重型 NLP 预处理

**Barcode First**:
条形码在 MVP 阶段是最高优先级识别信号。一旦条形码成功命中 `FoodItem`，本次识别链路应立即结束，不再继续 OCR 或图像识别，除非用户主动发起修正。
_Avoid_: 命中条码后继续多路识别校验

**Correction Signal**:
当用户纠正识别结果时，系统不仅要修正当前记录的绑定结果，还要留下可供后台治理的纠错信号。MVP 阶段即使不做复杂学习闭环，也应沉淀这些高价值反馈。
_Avoid_: 只修当前记录不留痕

**Location**:
MVP 阶段的地点信息只是 `FoodRecord` 上的自由文本补充，用于记录大致场景或店名，不作为独立实体建模。未来若需要门店点评、POI 搜索或同店分析，再单独引入结构化地点模型。
_Avoid_: Place, Store, POI（在 MVP 阶段）

**Price**:
MVP 阶段的价格信息属于 `FoodRecord`，表示本次事件中用户实际支付或主观感知的价格，不作为 `FoodItem` 的固定属性。未来若需要价格趋势或店铺比价，应从多条事件记录聚合。
_Avoid_: 固定商品标价, 平台统一参考价（在 MVP 阶段）
